# Accessibility — host API (Phase 3)

Android counterpart to webview2-gtk’s **structured host a11y** (not AT-SPI markdown).

See Windows: `/home/alan/git/webview2-gtk/docs/a11y.md` and `webview2gtk-host-api.h`.

Bug: [`docs/bugs/2026-07-23-a11y-walk-gtk-thread-anr.md`](bugs/2026-07-23-a11y-walk-gtk-thread-anr.md) — sync walk on GTK + IME `blockForMain` → ANR.

## Contract (locked to Windows shape)

```c
typedef struct wka_a11y_node {
  int id, parent_id;
  int x, y, w, h;   /* screen pixels (translation-adjusted) */
  char *name, *role, *value, *uri;
  gboolean can_invoke, can_set_value;
} wka_a11y_node;

gboolean wka_host_a11y_ensure (void);          /* Android-only force-on */
gboolean wka_host_a11y_walk (...);             /* sync — Android UI only */
gboolean wka_host_a11y_walk_foreach (...);     /* sync — Android UI only */
void wka_host_a11y_walk_foreach_async (...);   /* GLib/GTK callers — preferred */
gboolean wka_host_a11y_invoke (int id);
gboolean wka_host_a11y_set_value (int id, const char *utf8);
gboolean wka_host_a11y_focus (int id);
```

- **Ids** are dense walk indices; valid until the next walk.
- **Markdown / `^press:N`** stay in OLLMchat `libocwebkit` — not this library.
- **`AndroidAtspi`** Vala facade (parallel to `Win32Atspi` / GTK Atspi). The
  **checked-in** `vapi/webkitgtk-android-1.vapi` stays **GTK Atspi-shaped** (no
  Android-only methods). Meson subproject / `link_with` consumers (OLLMchat,
  this repo’s browser demo) use the **generated** library VAPI, which includes
  Android `refresh_async` for the UI-thread handoff.

## Caller contract (threads) — read this first

Pixiewood runs **GTK/GLib** and **Android UI** on different threads. Chromium WebView a11y (`getAccessibilityNodeProvider`, DirectConnection) must run on the **Android UI thread**. If GTK sync-waits on that work while Android UI is in `GlibContext.blockForMain` (e.g. IME), the app ANRs.

| Caller | Use |
|--------|-----|
| GLib / GTK / Vala (OLLMchat / demo) | **`using AndroidAtspi;` → `yield refresh_async();` → `get_desktop(0)`** then walk `Accessible` (same as Linux `Atspi` after refresh). |
| Already on Android UI (rare) | Sync host `wka_host_a11y_walk` / `ensure` OK. |
| Sync walk from GTK | **Refused** — empty result + `WebViewA11y` log (`refusing sync … deadlock risk`). |

**🚫** Do not wrap sync walk in a GLib worker and block on Android UI (`FutureTask` / latch) — same deadlock class as the ANR.

**🚫** Linux’s “AT-SPI on a worker thread” pattern alone is **not** sufficient on Android (wrong target thread).

**🚫** Do not add `refresh_async` to the **checked-in** public VAPI (keep that file GTK-shaped). It lives on `AndroidAtspi` in library sources for generated / subproject bindings — matching OLLMchat.

```vala
using AndroidAtspi;
yield refresh_async ();
var desktop = get_desktop (0);
/* … walk Accessible tree … */
```
## Force-on (app process — no TalkBack, no AccessibilityService)

Chromium WebView a11y is lazy. We force it from **app setup**, not by registering a service:

1. After WebView attach / page finish, `ensure` / `ensureAsync` runs on the UI thread (async walk also ensures on UI before collect).
2. **`AccessibilityNodeInfo.setQueryFromAppProcessEnabled(webView, true)`** (API 34+)
   on an **unsealed** node (`obtain` + `onInitializeAccessibilityNodeInfo`).
   That installs a ViewRootImpl **DirectConnection**, so
   `AccessibilityManager.isEnabled()` becomes true in-process.
3. Notify Chromium’s `AccessibilityStateChangeListener`s (DirectConnection alone
   may not fire them) so `WebContentsAccessibilityImpl` allows the provider.
4. Walk via `WebView.getAccessibilityNodeProvider()` (`HOST_VIEW_ID` root).

**🚫** No TalkBack. **🚫** No silent `AccessibilityService` “wake” helper.
